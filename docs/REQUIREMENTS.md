# AI人事24h 要件定義書 v7.1（最終確定版）

## 1. サービス概要

- **サービス名:** AI人事24h
- **種別:** B2B SaaS
- **概要:** AIが24時間365日、企業の一次面接を自動実施する。応募者はブラウザからリアルタイム音声AI面接を受け、企業は管理画面で録画・AI採点・要約を閲覧し、二次面接候補を選別する。
- **提供価値:** 面接工数削減、応募率向上（24時間対応）、統一基準による公平評価、録画・ログ保存による振り返り、テキストデータ永年蓄積によるデータ資産化、応募者への即時フィードバックによる面接体験向上。
- **関係者:** 応募者（Candidate）、企業ユーザー（Client）、運営ユーザー（Admin）の3者。

## 2. ビジネスモデル・課金

> 旧仕様（月額固定プラン ライト/スタンダード/プロ、31件目以降¥3,500の従量加算、自動繰上げ）は**廃止**。
> 現行は「従量課金（1面接・1人あたりの単価制）」。

### 2-1. 初期費用
- 20万円（税別）：導入セットアップ＋質問バンク設計含む
- 職種追加：+10万円/件（職種別に質問を変えたい場合）
- ※初期費用は月額固定プランとは独立した導入費用。旧プラン（ライト/スタンダード/プロ）とは無関係。

### 2-2. 料金体系（従量課金）
- **従量課金制**：請求額 = 当月の課金対象面接人数 × `companies.price_per_interview`（税別）
- 通常企業のデフォルト単価：**¥4,000 / 人（税別）**
- 月額固定プラン、31件目以降¥3,500の加算、自動繰上げは**使用しない**

### 2-3. 契約種別（plan）と単価（price_per_interview）
- `companies.plan` は **`pay_per_use`（通常）/ `custom`（特別契約）** の2値のみ（旧 light/standard/pro は使わない）
- `companies.price_per_interview`（整数・税別）が1人あたり単価
  - 通常企業（pay_per_use）：デフォルト 4,000
  - 特別企業（custom）：運営が任意設定（例 3,000）
- 特別契約（custom）は**運営管理画面でのみ設定**する裏側の契約種別
- **企業側画面には `custom`/カスタムプラン/特別契約/優遇プランの表記を一切出さない**
  - 企業側表示は「ご契約形態：従量課金」「ご契約単価：¥X,XXX / 人（税別）」のみ
- 企業側から `plan` / `price_per_interview` は変更不可（運営側のみ変更可）

### 2-4. 月間上限・翌月上限予約
- 月間上限は `companies.monthly_interview_limit`（**最低5人**）
- 上限到達後は新規面接の受付を**自動停止**
- 企業側は**翌月分の上限のみ予約変更可能**
  - `next_month_interview_limit`（予約値・最低5人）/ `next_month_limit_effective_month`（適用月）
  - 即時反映せず、**適用は必ず翌月1日**（`next_month_limit_effective_month` は翌月1日固定）
  - 使用済み人数より低い値も予約可能（翌月適用のため）
- 今月の上限・単価・契約種別は企業側から変更不可
- 運営側は企業ごとに「今月上限／翌月上限予約／契約種別（pay_per_use・custom）／単価／ステータス」を変更可能
- 月初昇格：適用月（翌月1日）到来後の最初のアクセス時に `next_month_interview_limit` を `monthly_interview_limit` へ反映し、予約2項目を null にクリア（二重反映なし）

### 2-5. 課金カウント基準
- 面接開始後**10分超**で1件カウント
- 10分未満は0件
- ネットワーク切断・サーバー接続エラーによる中断は0件
- 不適切行為による強制終了は面接時間に関わらず0件
- 面接開始＝AIが最初の音声を出力した時点のサーバータイムスタンプ
- 面接終了＝面接終了イベントのサーバータイムスタンプ
- **実装（Phase 2-a）**: `interviews.is_billable` は **end API（service-role）側で確定**（`duration_seconds > 600` ＝10分超で `true`、10分以下は `false`）。当月見込みは `is_billable=true` の件数 × 単価でリアルタイム算出。**確定請求（Stripe・billing_records writer / BATCH-001）は未実装**で、外部課金は最後にまとめて確認する。

### 2-6. 課金サイクル
- 月次・**月末締め**。日割りなし
- 当月請求額 = 当月の課金対象面接人数 × `price_per_interview`（税別）

### 2-7. 上限到達時
- 上限到達 → 新規応募者に「現在、面接の受付を一時停止しております。恐れ入りますが、しばらく経ってから再度お試しください。」と表示
- 企業には月間上限到達通知メール送信

### 2-8. 設定変更用パスワード（重要設定の保護）
- 重要設定の変更には、**ログインパスワードとは別の専用パスワード**を必須とする（いずれも hash 保存・平文保存禁止・ログインPW流用なし）
- 企業側：**管理者設定用パスワード**（`companies.company_setting_password_hash`）。翌月上限予約・CSV出力・重要設定変更時に使用
- 運営側：**運営管理設定変更用パスワード**（`admin_security_settings.setting_password_hash`、単一行運用）。企業ごとの今月上限/翌月予約/契約種別/単価/ステータス変更時に使用

## 3. ユーザー種別と権限

### 3-1. 応募者（Candidate）
- **認証:** SMS OTP認証（Twilio Verify）
- メールアドレスはフォーム入力のみ（認証には未使用）
- **再受験制限:** 同一電話番号×同一企業で1年間不可
- **重複検知:** 氏名＋生年月日の一致でフラグ通知（ブロックはしない）
- **冷やかしロック:** 面接開始後10分未満で離脱を3回で永久ロック（ネットワーク切断・サーバーエラー・面接開始前キャンセルは対象外）
- ロック解除は運営管理画面から手動

### 3-2. 企業ユーザー（Client）
- **認証:** メールアドレス＋パスワード（Supabase Auth）
- autocomplete="email" / autocomplete="current-password" 設定
- **セッション:** 有効期限24時間
- **ブルートフォース対策:** 5回連続失敗で15分ロック、10回連続失敗で永久ロック
- **データ分離:** Supabase RLSで自社データのみ閲覧可能
- 初回ログイン時にオンボーディングツアー表示（スキップ可能）

### 3-3. 運営ユーザー（Admin）
- **認証:** メールアドレス＋パスワード＋2FA（TOTP）、2FA必須
- 全企業のデータを横断閲覧可能
- 質問バンク設計・企業設定・テキストデータ永年保管・請求管理・停止管理・セキュリティアラート操作

## 4. 応募者側機能（F-C：Candidate Functions）

### F-C-001：画面A（開始・同意画面）
- 企業ごとのランダム文字列URLからアクセス
- Google reCAPTCHA v3設置
- 「この面接は最大約40分間です。静かな環境でお受けください。」案内文表示
- 2つの同意チェックボックス（両方チェックで次へ進行可能）
  - チェック1：応募先企業への映像・音声・回答内容提供同意（録画90日保存、テキスト永年保存）
  - チェック2：運営会社へのテキストデータ保管・匿名化活用同意
- プラン上限到達時は受付停止メッセージ表示

### F-C-002：画面B（SMS認証画面）
- 電話番号入力＋OTP入力（6桁、有効期限5分）
- 「面接をキャンセルする」リンク（確認ダイアログ→データ破棄→終了画面）
- **SMS認証4層防御:**
  - 第1層（OTP入力ロック）：5回連続失敗で永久ロック
  - 第2層（SMS送信制限）：同一電話番号1日5回、同一IP1時間3回、同一IPから異なる番号1日10回、再送間隔60秒
  - 第3層（reCAPTCHA）：SMS送信ボタンにreCAPTCHA v3
  - 第4層（異常検知）：大量リクエスト検知時に運営メール通知、IPブロックは運営手動
- 再受験チェック（同一電話番号×同一企業、1年以内）
- サポートメールアドレス表示（ドメイン取得後確定）

### F-C-003：画面C（履歴書相当フォーム）
- 「面接をキャンセルする」リンク
- **入力項目:**
  - 氏名（姓・名、必須）
  - フリガナ（セイ・メイ、必須）
  - 生年月日（日付ピッカー、年齢自動算出、必須）
  - 性別（プルダウン：男性/女性/その他/回答しない、必須）
  - 電話番号（SMS認証済み自動入力・編集不可、必須）
  - メールアドレス（必須）
  - 居住都道府県（47都道府県プルダウン、必須）
  - 最終学歴（プルダウン：中学卒/高校卒/専門学校卒/短大卒/大学卒/大学院卒/その他、必須）
  - 就業形態（新卒/中途、必須）
  - 業界経験（経験あり/未経験、必須）
  - 希望職種（企業ごとプルダウン、必須）
  - 職歴・業種（テキストエリア、上限500文字、任意）
  - 保有資格（テキストエリア、上限300文字、任意）

### F-C-004：画面D（デバイス準備・通信チェック画面）
- 「面接をキャンセルする」リンク
- マイク・カメラ使用許可取得
- **通信チェック合格基準:** WebRTC ICE取得成功、音声遅延1000ms以下、カメラ映像取得成功
- **マイクテスト:** 音声入力レベルメーター表示、閾値以上の音量検出で合格
- 全チェック合格で「面接を開始する」ボタン有効化

### F-C-005：画面E（リアルタイムAI面接画面）
- **AI:** OpenAI Realtime API（gpt-4o-mini-realtime-preview）、ブラウザWebRTC
- **最大40分**、沈黙2秒でVAD発話終了判定
- **画面構成:** 中央にAIアバター（静止画＋CSS/Canvasまばたき・微揺れ・脈動）、左上に応募者カメラ映像（小窓）
- **深掘り質問:** 回答が曖昧・矛盾・10秒未満の場合、最大1回/質問
- 35分経過時にAI口頭案内「残り時間が少なくなってきましたので、最後の質問に移ります」
- **回線品質監視:** WebRTC getStats()定期取得、閾値超過時に非侵入型バナー表示（面接は継続）
- **録画:** クライアント側MediaRecorder、WebM VP9、1280×720、30fps、音声ミックス
- **録画保存:** ハイブリッド方式（10秒間隔Blob蓄積＋3分ごとIndexedDBバックアップ）
- **ネットワーク切断:** 60秒以内で自動再接続（切断中もMediaRecorder継続）、60秒超で中断扱い
- **ブラウザ閉鎖:** 中断扱い、IndexedDB残存分のみ保存、再接続不可
- **無言1分:** 自動終了
- **不適切行為:** AI検知で即時終了、「不適切行為により終了」フラグ、課金なし
- **部分レポート:** 10分超で中断時、回答済み質問のみで部分評価レポート自動生成

### F-C-005-2：画面E2（送信中・フィードバック画面）
- プログレスバー＋「画面を閉じないでください」メッセージ
- Cloudflare R2へマルチパートアップロード（5MiB以上/パート、最大3回リトライ）
- beforeunloadで閉じ防止警告
- **フィードバック3ブロック（フェードインで順次表示）:**
  - ブロック1（即時）：面接サマリー（面接時間・質問数）、クライアント側情報のみ
  - ブロック2（10〜20秒後）：強みフィードバック（2〜3個、ポジティブのみ）、OpenAI GPT API
  - ブロック3（20〜40秒後）：パーソナリティ分析（行動力/協調性/分析力/創造性/安定性の5軸5段階レーダーチャート＋コメント）
- **3つの非同期処理が同時並行:** 録画アップロード・応募者フィードバック生成・企業向けレポート生成
- アップロード完了後 → IndexedDB削除 → 画面F遷移
- 部分欠損/録画なし時は運営メール通知

### F-C-006：画面F（終了画面）
- 「面接お疲れ様でした。結果は企業の担当者よりご連絡いたします。」
- 画面E2のフィードバック3ブロックをそのまま表示
- 満足度評価（1〜5段階の星評価、任意）、運営側で保管

### F-C-007：応募者ステータス仕様（表示導出・Phase 2-a 確定）
- **`applicants.status` はDB上 `準備中` / `完了` / `途中離脱` の3値のみ**（CHECK制約）。「面接中」は**applicants.status に保存しない**。
- **「面接中」は最新 `interviews.status='in_progress'` から画面表示上で導出**する（`applicants.status='完了'/'途中離脱'` は最優先表示）。
- `interviews.status` 運用 = `in_progress` / `completed` / `cancelled`。**`cancelled`→「途中離脱」、`completed`→「完了」**として扱う。
- 複数 interview がある場合は **最新 interview（created_at 降順先頭）の status を優先**（古い in_progress 孤児に引っ張られない）。

### F-C-008：公開面接フローと管理画面の分離（Phase 2-a 確定）
- 応募者向け公開フローは **`/interview/[slug]` 配下で完結**し、`/client/login` や `/admin` へ遷移しない。
- 終了画面（ended）は `/` や `/client/login` に飛ばさず、**応募者向け終了表示に留める**。
- 認証コード画面（verify）は誤コード時に**入力欄クリア＋先頭フォーカス＋インラインエラー**で再入力できる（認証は現状「1234」モック）。

### F-C-009：公開フローの書き込み方針（Phase 2-a 確定）
- **applicant作成 / interview開始 / 終了確定 / satisfaction / snapshot は service-role API 経由**（`POST /api/interview/[slug]/{applicant,start,end,satisfaction,snapshot}`）。applicants / interviews への **browser Supabase 直書きは撤去済み**。
- **ケイパビリティ・トークン**（HMAC-SHA256・`INTERVIEW_TOKEN_SECRET`）で slug / applicant_id / company / interview の整合を検証（詳細は API設計書）。
- **`applicants.status` の「完了」/「途中離脱」確定は end API（service-role）側で実施**（anon は RLS 上 applicants を更新できないため）。

## 5. 面接評価・レポート生成機能（F-R：Report Functions）

> **評価方式の正式名称：Evidence-based Competency Analysis（エビデンスベース・コンピテンシー分析）**
> 企業ごとに質問内容が異なっても、AIが面接全体の回答を横断的に分析し、発言内容・具体性・一貫性・行動傾向を根拠（evidence）に、6つの評価軸でコンピテンシーをスコア化する方式。各評価には根拠と信頼度（confidence）を付与し、判断材料が不足する軸は無理に断定せず「判断材料不足（insufficient）」として扱う。**質問ごとの軸事前設定（主軸/副軸/重み）は必須要件ではなく、企業のカスタム質問でも評価可能**。

### F-R-001：評価軸設計（全企業統一6軸）
全企業共通の6軸でコンピテンシーを評価する（軸キーは `communication / logical_thinking / initiative / desire / stress_tolerance / integrity`）。
1. **コミュニケーション力**（communication） — 質問の意図を正確に理解し、的確かつ分かりやすく回答できているか
2. **論理的思考力**（logical_thinking） — 回答に筋道が通っており、根拠や具体例を用いて説明できているか
3. **主体性・行動力**（initiative） — 自ら考え、行動した経験や姿勢が回答から読み取れるか
4. **志望度・意欲**（desire） — 応募企業・職種への関心や熱意が回答に表れているか
5. **ストレス耐性・柔軟性**（stress_tolerance） — 困難な状況や予想外の質問に対して冷静に対応できているか
6. **誠実性・一貫性**（integrity） — 回答に矛盾がなく、誇張や虚偽の兆候が見られないか

### F-R-002：採点方法（面接全体横断・根拠ベース）
- **面接全体横断評価:** AIは個々の質問単位ではなく、**面接全体の回答（全Q&A）を横断的に分析**して6軸を評価する。質問構成が企業ごとに異なっても、回答内容から該当軸の根拠を抽出してスコア化する。
- **軸ごとの出力:** 各軸に `score`（20点満点）／`rank`（A〜E）／`evidence`（応募者の原文引用＝根拠、1つ以上）／`confidence`（high / medium / low）／`insufficient_reason`（判断材料不足時の理由）を付与する。
- **判断材料不足の扱い:** 根拠が不足する軸は**無理に点数を作らず**、`score = null`（または低信頼）とし、`insufficient_reason` を持たせて「判断材料不足」として表示する。
- **総合スコア:** 判断可能な軸のスコアを集約し100点換算する（6軸合計120点満点を基準に、判断材料不足の軸の扱いは実装時に定義）。
- **質問ごとの軸事前設定は必須としない:** 旧方式（各質問に主軸/副軸/重みを運営が紐づけ加重平均）は廃止し、カスタム質問対応のため横断評価へ統一する。質問への軸ヒント付与は任意（評価精度向上の補助）。

| ランク | 軸スコア | 総合スコア |
|--------|----------|-----------|
| A | 17〜20点/軸 | 85〜100点 |
| B | 13〜16点/軸 | 70〜84点 |
| C | 9〜12点/軸 | 50〜69点 |
| D | 5〜8点/軸 | 30〜49点 |
| E | 1〜4点/軸 | 1〜29点 |

### F-R-003：軸ごとの評価出力
- 各軸ごとに：軸名、点数（20点満点／判断材料不足は null）、評価ランク（A〜E）、**根拠発言（evidence：応募者の原文引用）**、評価理由（2〜3文）、信頼度（confidence）、改善ポイント（D・E評価時）、判断材料不足時は `insufficient_reason`。
- 出力は `interview_results.evaluation_axes`（jsonb）に格納する（スキーマは API設計 INT-009 を参照）。

### F-R-004：レポート生成
- 面接終了後に非同期開始、録画アップロード完了を待たない
- **5分以内**の完了目標
- 完了時に企業へメール通知（管理画面リンク付き）
- 失敗時3回自動リトライ → 3回失敗で運営メール通知
- **レポート内容:** 質問ごとの採点一覧、Q&Aまとめ、要点（3〜7行）、総合評価レポート（強み・懸念点・深掘り推奨・総合所見）、全文ログ（AI/応募者区分）、録画動画

### F-R-005：部分レポート生成
- 10分超で中断時、回答済み質問のみで生成
- 「面接中断のため部分評価」フラグ付与
- 未回答質問は「未実施」表示

### F-R-006：録画保存
- Cloudflare R2 Privateバケット、**90日間保存**（ライフサイクルルール自動削除）
- 再生時は署名付きURL（有効期限10分、IP制限）、再生のたびに新規発行

## 11. 停止・再開

停止状態の正は **`companies.is_suspended`**（`companies.status` は admin一覧/詳細の表示で `status==='suspended'` を二次判定に使う補助項目であり、停止判定の正ではない）。面接受付ゲート（verify-url 等）は `is_suspended` のみで判定する。

- **通常停止（temporary）:** 企業申請（公開入力 `type:"normal"` → DB `request_type='temporary'`, `status='pending'`）→ 停止前は取り消し可能（`status='cancelled'`）→ BATCH-002 が `created_at + 1ヶ月` 経過で `companies.is_suspended=true` ＋ `suspension_requests.status='approved'`。
- **緊急停止（emergency）:** 企業申請（`request_type='emergency'`, `status='pending'`）→ 運営が承認/却下。承認時 `is_suspended=true` ＋ `status='approved'`、却下時 `status='rejected'`。
- **再開:** /admin/companies/[id] の「契約再開」で `is_suspended=false` ＋ `status='active'`（緊急承認・BATCH-002 で停止した企業も同様に復帰）。`suspension_requests` は履歴としてそのまま、`is_active` / `interview_url_active` は触らない。
- **強制停止:** /admin/companies/[id] の「契約停止」で `is_suspended=true` ＋ `status='suspended'`。

**suspension_requests スキーマ（実DB）:** `id, company_id, request_type, status, reason, created_at`。CHECK 制約は `request_type ∈ {temporary, emergency}` / `status ∈ {pending, approved, rejected, cancelled}`。`scheduled_stop_at` はDBカラムではなく `created_at + 1ヶ月` の導出値。`requested_by / cancelled_at / executed_at` 列は存在せず、`normal / pending_approval / executed` の値も使わない。
